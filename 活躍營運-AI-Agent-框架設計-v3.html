版本：v3.0 ｜ 日期：2026-03-29 ｜ 負責人：Jarvis

執行摘要

三層按需載入架構（Context → Playbook → Skill），每次對話僅消耗 ~1,700 tokens 指令層。
7 個 Playbook 涵蓋活躍組全部高頻工作流。12 個 Skill 可跨 Playbook 複用。
預估建置：15 個工作日（3 週），每日 4-6 小時。

四、Playbook 清單

PB-01活躍活動設計

觸發：需要規劃新的活躍活動
確認：活動目標 / 目標族群 / 上線時間 / IP或節慶主題
步驟：確認族群[SK-03] → 機制設計[SK-01] → 日曆排布[SK-08] → 獎勵設計[SK-04] → 文案[SK-05] → 整合企劃書
輸出：完整活動企劃書

PB-02指標異動分析

觸發：活躍指標波動超出 ±5%
確認：指標名稱 / 幅度 / 時間 / 市場 / 近期活動或版本變動
步驟：指標解讀[SK-02] → 分群分析[SK-03] → 假設樹[SK-07] → 應對方案
輸出：問題分析報告（問題→影響→根因假設→解方）

PB-03活躍週報產出

觸發：每週固定產出
步驟：指標解讀[SK-02] → 對照日曆節點[SK-08] → highlight + lowlight → 下週建議
輸出：週報 Markdown

PB-04活動上線審查

觸發：活動即將上線
步驟：獎勵驗證[SK-04] → 文案驗證[SK-05] → QA Checklist[SK-06] → 風險彙整
輸出：QA Checklist + 風險摘要

PB-05活動復盤

觸發：活動結束後 3-14 天
步驟：指標解讀[SK-02] → 分群差異[SK-03] → 競品對比[SK-09] → 假設樹[SK-07] → AB Test建議[SK-12] → 復盤報告[SK-11]
輸出：復盤報告（成果/分群差異/成功因素/失敗點/迭代計畫）

PB-06競品分析

觸發：需要了解競品活動策略
步驟：競品拆解[SK-09] → 機制對比[SK-01] → 獎勵對比[SK-04] → 差異化建議
輸出：競品分析報告

PB-07回流活動設計

觸發：需要設計流失/風險玩家回流策略
步驟：族群定義[SK-03] → 機制設計[SK-01] → 獎勵設計[SK-04] → 推播序列[SK-10] → 文案[SK-05] → AB Test[SK-12]
輸出：回流方案（族群/機制/獎勵/推播序列/AB Test）

五、Skill 清單與 Few-shot

SK-01活動機制設計

用途：根據目標與族群，產出 3 個機制選項並比較 trade-off

Few-shot：
輸入：目標：提升 D7 Retention｜族群：D4-14 早期活躍玩家｜天數：7 天｜無 IP

輸出：
A. 7 日任務串：每日完成 1 模式任務→累積解鎖 D3/D5/D7 階梯獎勵｜⭐⭐⭐⭐ engagement｜低複雜度
B. 新手排位衝刺：7 天打 20 場排位→前 30% 獲限定掛件｜⭐⭐⭐ 排斥休閒玩家｜低複雜度
C. 好友組隊挑戰：邀 1 好友組隊打 5 場→雙方獲獎｜⭐⭐⭐⭐ 社交黏著｜中複雜度

建議：A 主體 + C 支線。B 不適合此族群（早期玩家排位能力弱，挫敗風險高）。

SK-02指標解讀框架

用途：結構化解讀活躍指標，產出判斷而非轉述數字

Few-shot：
輸入：D7 Retention 本週 18.2%（台灣）、上週 21.5%、無活動變動

輸出：🔴 D7 Retention 下滑 3.3pp（-15%），幅度超出正常波動。無活動節點可解釋，且為台灣單一市場現象。建議排查：(1) 近期版本更新是否引入台灣地區 bug；(2) UA 投放渠道結構變化導致新用戶品質下降。

SK-03玩家分群與定向

用途：根據任務目標識別目標族群，給出行為特徵與觸達策略

Few-shot：
輸入：設計一個提升 D7 Retention 的活動

輸出：目標族群：早期活躍期（D4-14）
行為特徵：已完成教程，週登入 3-4 天，社交功能使用率低
觸達：遊戲內彈窗 + 推播（D4/D6/D7）
注意：任務難度需低於核心玩家標準，避免高門檻機制

SK-04獎勵結構設計

用途：設計獎勵的品項、數量、發放節奏與稀缺感

Few-shot：7 天登入活動 → 早期活躍玩家

D1: 500幣+經驗卡（低門檻進門）
D2: 普通掛件（視覺獎勵開始）
D3: 1000幣（加倍強化慣性）
D4: 皮膚碎片×3（沉沒成本鉤子——需5個合成）
D5: 皮膚碎片×2→可合成（達成感 peak）
D6: 限時武器試用卡（製造付費動機）
D7: 限定頭像框（稀缺性收尾）

節奏：D1-3 建慣性、D4 設鉤子、D5 達成感、D6 付費轉化、D7 稀缺收尾

SK-05活動文案產出

用途：產出活動標題、說明、推播文案（可直接用）

Few-shot：7天新兵集結｜每日完成指定模式→累積獎勵｜限定皮膚｜台灣

標題：新兵集結令——連續 7 天，解鎖限定裝備
說明：每日完成指定模式任務，累積天數解鎖階梯獎勵。堅持 7 天，限定皮膚直接入手。
推播 D1：「新兵集結！7天任務已開啟，限定皮膚等你拿」
推播 D3：「你已完成 3 天任務，限定皮膚只差 4 天」
推播 D6：「最後 1 天！完成今日任務即可解鎖限定皮膚」

SK-06QA Checklist 生成

用途：根據活動類型生成上線前檢查清單

固定 7 項 + 按活動類型追加（登入/排位/公會/IP 聯動各有額外檢查項）。輸出為表格：檢查項/預期/實際/狀態/備註。

SK-07假設樹產出

用途：針對異動產出 MECE 假設樹，每個假設附驗證方法

Few-shot：台灣 D7 Retention 下滑 3.3pp，無活動變動

D7 Retention 下滑 3.3pp
├── A. 新用戶品質（UA端）
│   ├── A1. 渠道 mix 變更 → 比對渠道結構
│   ├── A2. 新素材引入低意圖用戶 → 按素材分組比較
│   └── A3. 自然/付費量比例變化 → 拉分組D7
├── B. 遊戲體驗（產品端）
│   ├── B1. 版本 bug → 查客服工單量
│   ├── B2. 新手引導異常 → 查 Tutorial 完成率
│   └── B3. 配對不均 → 查新手勝率分布
└── C. 外部因素
    ├── C1. 競品搶量 → 查競品活動時間
    └── C2. 假期/考試季 → 查歷史同期

SK-08事件日曆排布

用途：評估活動在月度日曆中的位置，避免過載或空檔

Few-shot：4月日曆有 4/1-14 賽季活動、4/21-28 公會聯賽，想加 7 天登入活動

建議 4/15-4/21。賽季 4/14 結束後有 DAU 回落，此時段填補空檔。4/21 與公會聯賽重疊 1 天可接受。不建議 4/7-13（與賽季重疊，任務過載）。

SK-09競品活動拆解

用途：結構化拆解競品活動（目標/族群/機制/獎勵/可借鑑點/差異化）

Few-shot：PUBG Mobile「Evangelion 聯動」3 週

目標推測：拉新+DAU（EVA IP 吸引動漫受眾）
機制：主線 8 關故事任務 + 每日登入 + 聯動抽獎池
獎勵：免費 EVA 頭像框/塗裝；付費抽 EVA 角色皮膚
可借鑑：IP 聯動做完整故事線→提升沉浸感
差異化：PUBG 是單人 PVE，CODM 可做多人合作 PVE 任務線

SK-10推播策略設計

用途：設計推播觸發規則、時機、文案與頻率控制

Few-shot：回流推播，族群：7-14 天未登入

Day 7：「你錯過了新地圖 XX 的首次挑戰」（內容驅動）
Day 10：「你的戰隊夥伴 [好友名] 正在等你上線」（社交驅動）
Day 14：「回來就送 500 CP + 限定武器試用卡（限 48h）」（獎勵+時間壓力）

頻率：7 天內最多 3 則，超過不再推播。

用途：結構化輸出復盤報告（摘要/指標/分群差異/成功因素/失敗點/迭代行動）

SK-12A/B Test 設計

用途：設計 AB Test 方案（假設/Control vs Treatment/指標/樣本/成功標準）

Few-shot：假設：D4 獎勵改為皮膚碎片可提升 D4→D5 留存

Control：D4 = 500 幣（現行）
Treatment：D4 = 皮膚碎片×3（需 5 個合成）
主指標：D4→D5 繼續參與率
次指標：D7 完成率、活動 DAU

六、寫法原則（好壞案例）

L1Agent Context

寫什麼：角色定位、職責邊界、輸出對象、遇到模糊情況的預設行為。
不寫什麼：任務步驟、輸出格式細節、執行邏輯——這些屬於 L2/L3。

✅ 好案例

## 活躍營運 AI Agent

你是 CODM 活躍組的資深執行夥伴。你的輸出對象是活躍組同事，不是主管或開發商。

**職責範圍：**
- 設計與優化活躍活動（任務機制、獎勵、時程）
- 解讀活躍指標並提出應對方案
- 產出活動企劃書、週報分析、上線審查、活動復盤、競品分析、回流策略

**邊界（不碰）：**
- 付費相關定價與排期（屬商業化）
- 產品功能開發與 bug 修復（屬產品組）
- 跨市場發行策略對齊（屬 Producer 層級）

**遇到模糊情況：**
- 缺少必要資料時，先列出「我需要以下資訊才能繼續」，不要猜測
- 有多個合理方向時，給出選項讓使用者決定，不要自行選擇
為什麼好：明確定義「輸出對象」→ 語氣調整為同事層級，不是彙報口吻；邊界寫的是「不碰什麼 + 為什麼」→ 不會越界；預設行為有明確指令 → 遇到模糊情況有一致的處理方式。

❌ 壞案例

你是一個活躍營運 AI 助手，幫助團隊做活動設計和數據分析。
你要友善、專業、有效率。
有任何問題請回答。
為什麼壞：「友善、專業、有效率」沒有可操作的行為指令，Agent 無法依此做判斷；沒有邊界定義 → 遇到付費問題會嘗試回答；沒有輸出對象 → 語氣和格式會飄移；沒有模糊情況規則 → Agent 自行猜測，出錯後你不知道為什麼。

L2Playbook

寫什麼：觸發情境、執行前需確認的資訊、步驟順序（每步一行）、輸出物定義。
不寫什麼：每個步驟的具體執行細節——那是 Skill 的工作。

✅ 好案例

## PB-01：活躍活動設計

**觸發情境：** 使用者需要規劃一個新的活躍活動

**執行前確認（缺少時主動詢問）：**
- 活動目標（提升 DAU / 改善 D7 Retention / 拉回流失用戶）
- 目標玩家族群（全玩家 / 核心玩家 / 流失玩家）
- 活動上線時間與持續天數

**執行步驟：**
1. 確認目標玩家族群 → [呼叫 SK-03]
2. 設計活動機制選項 → [呼叫 SK-01]
3. 安排活動在月度日曆中的位置 → [呼叫 SK-08]
4. 設定獎勵結構與發放節奏 → [呼叫 SK-04]
5. 產出活動文案草稿 → [呼叫 SK-05]
6. 整合輸出完整活動企劃書

**輸出物：** 活動企劃書（目標 / 族群 / 機制 / 獎勵 / 時程 / 預期 impact / 風險）

**成功標準：** 輸出可直接送審，不需要使用者大幅補充或修改
為什麼好：「執行前確認」明確列出缺少哪些資訊時要問 → 不會在資訊不足時就輸出；每步只有一行，細節全部委託 Skill → Playbook 保持精簡；「成功標準」讓 Agent 有自我驗證的基準。

❌ 壞案例

## 活動設計流程

當使用者要設計活動時，你需要：
1. 先了解活動目標是什麼
2. 幫他想一些好的活動機制，例如每日登入獎勵、累積任務、限時挑戰等，
   要考慮玩家的參與意願和執行難度，以及獎勵設計要有吸引力但不能太貴，
   要符合遊戲的經濟體系，同時也要考慮文案的清晰度...
3. 輸出企劃書
為什麼壞：步驟 2 把機制設計、獎勵設計、文案三件事混在一起 → 這是三個不同 Skill 應該分別處理的事；沒有「執行前確認」→ 資訊不足時 Agent 自行猜測；細節描述每次都要載入，即使某個環節這次用不到。

L3Skill

寫什麼：輸入需求、執行邏輯、輸出格式、few-shot 範例（品質最大的槓桿）、常見錯誤。
不寫什麼：任務背景、整體目標——那是 Playbook 的工作。

✅ 好案例

## SK-02：指標解讀框架

**輸入需求：** 指標名稱、當期數值、對比基準（上週 / 上月 / 歷史均值）、時間區間

**執行邏輯：**
1. 計算變動幅度（%）並判斷是否超出正常波動範圍（±5% 為正常，超過需標記）
2. 對照活動節點：異動是否與活動上線 / 結束時間吻合？
3. 跨市場比較：是全市場現象還是單一市場
4. 產出解讀結論（結論先行，數字後補）

**few-shot 範例：**
輸入：D7 Retention 本週 18.2%（台灣），上週 21.5%，無活動節點變化
輸出：🔴 D7 Retention 下滑 3.3pp（-15%），幅度超出正常波動範圍，且無活動結束可解釋，
      且為台灣單一市場現象。建議優先排查：(1) 近期版本更新是否引入台灣地區 bug；
      (2) UA 投放渠道結構變化導致新用戶品質下降。

**輸出格式：** 每個指標 2-4 行，結論先行，數字後補，標記異常等級（⚠️ 需關注 / 🔴 緊急）

**常見錯誤：** 不要只轉述數字（「本週 DAU 為 X，上週為 Y」），必須給出判斷（正常/異常）和建議方向。
為什麼好：few-shot 範例直接示範「好的輸出長什麼樣」→ 大幅提升穩定性；「常見錯誤」明確告訴 Agent 不要做什麼 → 避免輸出退化；輸出格式有明確長度預期。

❌ 壞案例

## 指標分析

幫使用者分析活躍指標的表現，包括 DAU、Retention、在線時長等。
要給出有意義的分析，找出問題並提供建議。
為什麼壞：沒有輸入需求定義 → 不知道要先問什麼資料；沒有執行邏輯 → 每次輸出結構不同；沒有 few-shot 範例 → 品質完全依賴模型泛化能力，不穩定；「有意義的分析」是主觀描述，對 Agent 毫無指導作用。
